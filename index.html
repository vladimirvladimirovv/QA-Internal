01. QA Philosophy

Risk Mitigation > Bug Finding

QA is the bridge between business requirements and technical execution. My goal isn't just to report that something is broken, but to evaluate the Risk Impact on the user and the revenue stream.

Think like a User, Test like a Breaker

Empathy for the user drives functional testing; technical curiosity drives edge-case discovery. I advocate for Shift-Left, ensuring QA is present during the design phase to prevent bugs before they are coded.

02. Testing Strategy

Sanity & Smoke

Rapid validation of core business flows (Login, Checkout, Profile) after deployment to ensure the environment is stable.

Regression

Automated and manual verification that new changes haven't degraded existing functionality, focused on high-risk areas.

Exploratory

Unstructured, time-boxed sessions to discover edge cases that scripted tests might miss, mimicking "wild" user behavior.

03. API Guidelines (Postman)

Code Category Vladimir's QA Expectation
200/201 Success Check payload matches schema and data persists in DB via DBeaver.
400 Bad Request Validation errors must be meaningful and localized, not generic.
401/403 Auth Verify JWT expiration and RBAC (Role-Based Access Control).
500/503 Error Failure must be logged in Grafana with a unique TraceID for debugging.

04. The Investigation Loop

1. Frontend Validation

Inspect the browser console (F12). Is it a UI crash or a failed network call? Identify the payload and request headers.

2. API Isolation (Postman)

Replicate the request in Postman. Does the endpoint fail outside the UI? This isolates the bug to either Frontend or Backend.

3. Observability (Grafana)

Search for the Trace ID or User ID in Grafana logs. Look for Stack Traces or DB connection timeouts.

4. Persistence Check (DBeaver)

Run a SQL query to check the DB state. Did the API claim it saved the data, but the DB remains unchanged? (Silent Failure).

05. Bug Reporting Standards

Severity vs Priority

  • Critical: System crash, data loss, security breach. (Must fix now).
  • High: Major feature broken with no workaround.
  • Medium: Feature bug with a reasonable workaround.
  • Low: UI polish, typos, or minor visual issues.

Common Mistakes to Avoid

  • ❌ Vague Titles: "Button doesn't work"
  • ❌ Missing Steps: "Try to login" (Need credentials/env)
  • ❌ No Evidence: Reports without logs or screenshots.

08. Release Traceability

Releases are not just code deployments; they are audit trails. Every release note must include:

Features

Newly added capabilities with linked Jira tickets.

Fixes

Resolved bugs and performance improvements.

QA Verdict

Sign-off status, known issues, and rollback plan.
